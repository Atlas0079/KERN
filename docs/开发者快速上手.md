# 开发者快速上手

本文档面向第一次接触本项目的开发者，目标是在本地完成一次可运行的模拟。

## 1. 环境准备

1. 安装 Python 3.10+（建议使用虚拟环境）。
2. 进入项目根目录。
3. 安装依赖：

```bash
pip install -r requirements.txt
```

## 2. 选择启动配置

仓库已提供两个可直接使用的配置文件：

- `runtime_config.farm.json`
- `runtime_config.werewolf.json`

它们都使用 `{"env": {...}}` 结构，入口脚本会读取其中字段作为运行参数。

## 3. 首次运行（推荐先跑无 LLM 场景）

运行：

```bash
python app.py --config runtime_config.werewolf.json
```

运行后会看到日志输出，流程包括：

1. 读取并解析配置文件
2. 加载 `Data/` 下 world / recipes / reactions / entities
3. 执行数据校验并构建世界状态
4. 进入 tick 循环并持续输出事件日志
5. 在 checkpoint 目录写入快照文件

## 4. 运行 Farm 场景

```bash
python app.py --config runtime_config.farm.json
```

如果该配置中 `USE_LLM=1`，程序会初始化 LLM provider 并在决策阶段调用模型。

## 5. 使用自定义配置

### 5.1 从样例复制

复制 `runtime_config.example.json` 为你的运行文件，例如 `runtime_config.local.json`，然后按需修改 `env` 字段。

### 5.2 启动时指定

```bash
python app.py --config runtime_config.local.json
```

### 5.3 通过环境变量指定

当不传 `--config` 时，可用环境变量指定配置文件：

```powershell
$env:RUNTIME_CONFIG_FILE="runtime_config.local.json"
python app.py
```

## 6. LLM 模式最小检查项

当 `USE_LLM=1` 时，至少确认以下字段已设置为有效值：

- `LLM_PROVIDER`
- `LLM_BASE_URL` / `GEMINI_BASE_URL`（取决于 provider）
- `LLM_API_PREFIX` / `GEMINI_API_PREFIX`
- `LLM_API_KEY` / `GEMINI_API_KEY`
- `LLM_PLANNER_MODEL`
- `LLM_GROUNDER_MODEL`

## 7. 常用运行参数

可以在运行配置文件 `env` 中直接调整：

- `MAX_TICKS`：本次运行的总 tick 上限
- `LOG_LEVEL`：日志级别
- `LOG_CATEGORIES`：日志分类过滤
- `CHECKPOINT_DIR`：checkpoint 输出目录
- `CHECKPOINT_RESTORE_FILE`：从指定 checkpoint 恢复
- `CHECKPOINT_RESTORE_DIR`：从目录中最新 checkpoint 恢复
- `WORKFLOW_CONTRACT_ON_ERROR`：workflow 契约错误策略，`fail_fast` 或 `degrade_to_noop`

## 8. Workflow 插件契约（Agent 决策）

当前 Agent 流程使用“可插拔 workflow”契约。workflow 只读访问 `ws`，输出一个标准字典，由框架统一分发到 Executor。
默认实现位于 `newserver/agent_workflow/` 目录。

接口约定：

```python
decision = workflow.decide(ws_view, recipe_db, actor_id, reason, mode_context)
```

决策字典三种类型：

- `{"type":"apply_commands", "commands":[{"verb":"...", "target_id":"...", "parameters":{...}}]}`（推荐，执行层会编译为 effects）
- `{"type":"apply_effects", "operations":[{"effect": {...}, "context": {...}}]}`（兼容）
- `{"type":"noop"}`
- `{"type":"error", "error":{"kind":"temporary|business|contract","code":"...","message":"..."}}`

说明：

- `apply_commands` 由执行层统一通过 `InteractionEngine` 编译为 effects，再由 `Executor` 落地。
- `recipe_db` 作为独立输入显式传入 workflow，不再通过 `ws_view` 隐式注入。
- `apply_effects` 的 `operations` 会按顺序逐条交给 `Executor` 执行。
- `contract` 级错误会触发 `WORKFLOW_CONTRACT_ON_ERROR` 策略。
- 默认内置 workflow（LLM/SimplePolicy）都已使用该契约。

## 9. 结果文件位置

默认情况下，checkpoint 输出目录为：

`checkpoints/<WORLD_JSON>/`

目录中会出现：

- `tick_000000.json`、`tick_000001.json` 等快照
- `simulation_log.json` 全局日志
