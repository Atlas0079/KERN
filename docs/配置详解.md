# 配置详解

本文档说明当前代码中实际生效的运行配置项与加载逻辑，覆盖入口脚本 `app.py` 及其调用链路。

## 1. 启动与配置加载

### 1.1 启动命令

```bash
python app.py --config runtime_config.farm.json
```

`--config` 为可选参数，未提供时会继续按以下顺序查找配置文件：

1. `--config` 指定路径
2. 环境变量 `RUNTIME_CONFIG_FILE`
3. 默认文件 `runtime_config.json`

### 1.2 runtime_config 文件格式

运行时配置文件必须是 JSON 对象，并包含 `env` 对象：

```json
{
  "env": {
    "USE_LLM": "1",
    "WORLD_JSON": "Farm/World.json"
  }
}
```

`env` 中所有值会被转换为字符串后参与后续解析。

### 1.3 布尔和整数解析规则

- 布尔真值：`1`、`true`、`yes`、`on`（大小写不敏感）
- 其他字符串按 `False` 处理
- 整数字段若解析失败，会回退到代码中的默认值

## 2. 数据装载配置

以下字段由 `app.py` 读取，并传入 `newserver.data.loader.load_data_bundle`：

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `WORLD_JSON` | `World.json` | 世界文件路径，基于 `Data/` 目录 |
| `RECIPES_JSONS` | `Recipes.json` | 配方文件列表，逗号分隔 |
| `REACTIONS_JSONS` | `Reactions.json` | 反应规则文件列表，逗号分隔 |
| `ENTITIES_DIRS` | `Entities` | 实体模板目录列表，逗号分隔 |

### 2.1 路径规则

- `load_data_bundle` 会在项目根目录下查找 `Data/`。
- 若传入目录本身就是 `Data`，则直接使用该目录。
- `RECIPES_JSONS` 与 `REACTIONS_JSONS` 中不存在的文件会被跳过。
- `ENTITIES_DIRS` 下所有 `*.json` 会按文件名排序加载，后加载条目会覆盖同名模板。

## 3. 校验与世界构建

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `VALIDATION_MODE` | `fast` | 数据校验模式，支持 `off` / `fast` / `strict` |

流程说明：

1. 未使用 checkpoint 恢复时，先执行 `validate_bundle`。
2. 校验通过后执行 `build_world_state` 构建初始世界状态。
3. 校验失败时进程会抛出异常并退出。

## 4. 运行循环与调度配置

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `USE_LLM` | `0` | 是否使用 LLM 行为提供器 |
| `MAX_TICKS` | 空 | 显式设置最大 tick 数 |
| `MAX_TICKS_DEFAULT_LLM` | `15` | `USE_LLM=1` 且未设置 `MAX_TICKS` 时使用 |
| `MAX_TICKS_DEFAULT_NO_LLM` | `65` | `USE_LLM=0` 且未设置 `MAX_TICKS` 时使用 |
| `MAX_TRIGGER_DEPTH` | `4` | 反应链递归深度上限 |
| `DIALOGUE_BUDGET_LIMIT_PER_LOCATION` | `4` | 每个地点每 tick 的对话预算 |
| `DIALOGUE_LOG_FULL` | `0` | 对话日志详细开关 |
| `DEMO_DURATION_TEST` | `0` | 是否执行演示性 Wait 命令段落 |

`MAX_TICKS` 的优先级高于两个默认值字段。

## 5. LLM 配置

当 `USE_LLM=1` 时，`build_default_llm_provider` 会读取下列字段。

### 5.1 通用字段

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `LLM_PROVIDER` | `openai_compat` | 提供器类型，`openai_compat` 或 `gemini` |
| `LLM_TIMEOUT_SECONDS` | `60` | 单次请求超时（秒） |
| `LLM_MAX_RETRIES` | `2` | 失败重试次数 |
| `LLM_RETRY_BACKOFF_SECONDS` | `1.0` | 重试退避时间（秒） |
| `LLM_PLANNER_MODEL` | 提供器相关默认值 | 规划模型名 |
| `LLM_GROUNDER_MODEL` | 提供器相关默认值 | 落地模型名 |
| `LLM_FOCUS_AGENT_ID` | 空 | 聚焦某一 agent 的日志 |
| `LLM_FOCUS_LOG_PROMPTS` | `0` | 是否输出 prompt 日志 |
| `LLM_FOCUS_LOG_PERCEPTION` | `1` | 是否输出感知日志 |
| `LLM_FAILURE_THRESHOLD` | `3` | 连续失败阈值 |
| `LLM_FAILURE_COOLDOWN_TICKS` | `60` | 冷却 tick |
| `LLM_DEBUG_VIEW` | 空 | 调试视图标志 |

### 5.2 `openai_compat` 提供器字段

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `LLM_BASE_URL` | `https://api.aabao.top` | 基础地址 |
| `LLM_API_PREFIX` | `/v1` | API 前缀 |
| `LLM_API_KEY` | `REPLACE_ME` | API Key |
| `LLM_PLANNER_MODEL` | `gemini-3-pro-preview` | planner 默认模型 |
| `LLM_GROUNDER_MODEL` | `gemini-3-flash-preview` | grounder 默认模型 |

### 5.3 `gemini` 提供器字段

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `GEMINI_BASE_URL` | `https://generativelanguage.googleapis.com` | 基础地址 |
| `GEMINI_API_PREFIX` | `/v1beta` | API 前缀 |
| `GEMINI_API_KEY` | `REPLACE_ME` | API Key |
| `LLM_PLANNER_MODEL` | `gemini-1.5-pro` | planner 默认模型 |
| `LLM_GROUNDER_MODEL` | `gemini-1.5-flash` | grounder 默认模型 |

## 6. 日志配置

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `LOG_LEVEL` | `info` | 日志级别：`trace`/`debug`/`info`/`warn`/`error` |
| `LOG_CATEGORIES` | `*` | 分类过滤，逗号分隔，`*` 表示全开 |
| `LOG_JSON` | `0` | 是否输出 JSON 日志 |
| `LOG_BUFFER_SIZE` | `1000` | 内存缓冲条数上限 |

日志系统由 `newserver.log_manager.configure_logger` 初始化。

## 7. Checkpoint 配置

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `CHECKPOINT_EVERY_TICK` | `1` | 每 tick 保存 checkpoint 开关 |
| `CHECKPOINT_INCLUDE_LOGS` | `1` | checkpoint 文件中包含当前 tick 日志 |
| `CHECKPOINT_DIR` | `checkpoints/<WORLD_JSON>` | 输出目录 |
| `CHECKPOINT_RESTORE_FILE` | 空 | 指定恢复文件 |
| `CHECKPOINT_RESTORE_DIR` | 空 | 指定恢复目录，自动取最新 `tick_*.json` |

### 7.1 恢复优先级

1. 若 `CHECKPOINT_RESTORE_FILE` 非空且文件存在，使用该文件恢复。
2. 否则若 `CHECKPOINT_RESTORE_DIR` 指向有效目录，使用该目录中最新 `tick_*.json`。
3. 否则执行“校验 + 世界构建”冷启动流程。

### 7.2 运行期输出

启用 checkpoint 时，`WorldManager` 会写入：

- `tick_000000.json`、`tick_000001.json` 等逐 tick 快照
- 同目录 `simulation_log.json` 全局日志文件

## 9. TaskPolicy（任务中断策略）

`TaskPolicy` 不属于 `runtime_config.env`，而是定义在配方（recipe）中，用于控制任务被中断/取消时的行为。

当前实现读取顺序：

1. `recipe.process.task_policy`（推荐）
2. `recipe.task_policy`（兼容）

创建任务时，策略会被固化到 `task.parameters.task_policy`。

### 9.1 字段说明

| 字段名 | 默认值 | 说明 |
| --- | --- | --- |
| `interrupt_mode` | `pause_keep_progress` | 中断模式：`forbidden` / `pause_keep_progress` / `pause_reset_progress` / `cancel` / `fail` |
| `allow_voluntary_interrupt` | `true` | 是否允许 agent 主动中断 |
| `allow_voluntary_cancel` | `true` | 是否允许 agent 主动取消 |

兼容别名（会被标准化）：

- `pausable` -> `pause_keep_progress`
- `restartable` -> `pause_reset_progress`
- `cancellable` -> `cancel`
- `fail_on_interrupt` -> `fail`

### 9.2 配方示例

```json
{
  "id": "recipe.travel",
  "verb": "Travel",
  "process": {
    "required_progress": 3,
    "task_policy": {
      "interrupt_mode": "pause_keep_progress",
      "allow_voluntary_interrupt": true,
      "allow_voluntary_cancel": false
    }
  },
  "outputs": [
    { "effect": "MoveEntity", "target": "self", "to": "param:to_location_id" }
  ]
}
```

### 9.3 对感知层的暴露（最小摘要）

为避免把过多内部策略细节暴露给 agent，当前只提供：

- `current_task_interrupt_mode`
- `current_task_can_interrupt`
- `current_task_can_cancel`
