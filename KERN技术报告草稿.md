# KERN 技术报告

## 1. 摘要

KERN（Knowledge, Environment, Runtime, Narrative）是一个基于 ECS（Entity-Component System）与数据驱动的离散事件仿真（Discrete-Event Simulation, DES）沙盒内核。它面向多智能体与大语言模型实验，目标是提供一个可配置、可复用、可审计的运行基座。

KERN 试图解决的问题是：在许多实验系统中，场景定义、规则执行、状态更新与智能体工作流往往被直接耦合在一起。这样一来，研究者一旦需要切换场景、调整规则或更换智能体，就必须改动核心逻辑，系统很难复用，也很难稳定迭代。

为了解决这一问题，KERN 采用了三项核心设计：一是使用 ECS 组织世界状态，使实体能力由组件组合表达；二是将交互规则与状态变化表达为数据驱动的声明式机制；三是将所有世界写操作统一收口到执行器，确保状态更新具备一致的执行入口。

当前版本已经实现了配置加载、数据校验、世界构建、Tick 调度、规则执行、日志记录、Checkpoint 保存与恢复，以及基础的外部智能体接入链路。当前版本的主要边界在于：展示层仍偏向调试观察工具，智能体工作流仍属于基线实现，尚不构成系统的核心贡献。

## 2. 问题定义

### 2.1 多智能体实验中的工程问题

在传统多智能体实验中，场景结构、交互逻辑与状态更新流程往往被直接硬编码。这种做法在小规模原型阶段可行，但当研究者需要频繁调整地点布局、任务结构、交互规则或智能体策略时，问题会迅速暴露。

具体而言，这类系统通常存在三类工程问题：

1. 场景定义与核心逻辑耦合：新增地点、实体或规则时，往往需要同步修改运行时代码。
2. 状态写入入口分散：世界状态可能在多个逻辑分支中被直接修改，导致行为难以复盘，错误难以定位。
3. 智能体接入成本高：环境主循环通常默认绑定某一种固定工作流，替换新的决策模块会牵连已有执行链。

因此，系统需要一个更稳定的基础架构，使“世界定义”“规则匹配”“状态结算”和“智能体接入”彼此分离，同时仍能在同一个运行时里形成闭环。

### 2.2 KERN 的目标问题

KERN 关注的不是单一场景，也不是单一智能体算法，而是一个更基础的问题：如何构建一个可以被持续复用的离散仿真沙盒内核。这个内核需要满足以下要求：

- 世界状态能够通过数据定义，而不是通过类继承层层硬编码。
- 状态更新必须有统一入口，便于审计、复盘和回放。
- 规则可以声明式扩展，而不需要不断扩大主循环的条件分支。
- 智能体可以替换，但底层物理结算链应保持稳定。

## 3. 设计目标

围绕上述问题，KERN 在当前版本中采用了四个明确的设计目标。

### 3.1 数据驱动的世界定义

目标是让研究者通过 JSON 配置而不是直接修改核心代码来构建场景。当前系统中的地点、路径、实体模板、交互配方与反应规则均由配置描述。例如，一个“苹果”实体不需要单独编写代码类，只需在 JSON 中组合 `EdibleComponent` 与 `TagComponent` 即可完成定义。

### 3.2 统一的状态写入入口

目标是避免世界状态在不同逻辑分支中被随意修改。KERN 将所有状态写操作统一表达为 `effect` 指令，再交由执行器落地。这种做法牺牲了少量的执行效率与配置简洁性，但换来了更清晰的执行边界和更稳定的审计能力。

### 3.3 可插拔的智能体接入

目标是将智能体工作流与仿真主循环解耦。KERN 不预设某一种固定的智能体内部实现，而是将输入输出边界固定为“感知世界状态”和“输出动作命令”。这样做的原因是，内核需要长期稳定，而智能体工作流本身会持续迭代。

### 3.4 可复用的场景与验证框架

目标是让不同复杂度的场景共享同一套执行链。换言之，场景内容可以变化，但底层的 Tick 推进、规则匹配、Effect 执行与日志记录机制应保持一致，以便系统能够在同一内核上复用不同的实验环境。

## 4. 核心抽象

### 4.1 WorldState 与 ECS

KERN 的运行时状态统一收束在 `WorldState` 中。它保存实体、地点、路径、任务以及与时间推进相关的运行信息，是整个仿真沙盒的核心状态容器。

KERN 采用 ECS（Entity-Component-System，实体-组件-系统）组织世界数据。在这一设计中：

- **实体（Entity）** 是一个唯一 ID，本身不承载复杂逻辑。
- **组件（Component）** 是纯数据结构，用于表达生命值、标签、容器属性、状态等能力。
- **系统（System）** 是处理这些数据的逻辑单元，例如触发器系统、感知系统与执行系统。

这种设计的原则是：通过组件组合定义能力，而不是通过继承层级定义角色。这样做可以降低实体类型膨胀的问题，也更适合数据驱动配置。

### 4.2 Recipe、Reaction、Effect 与 Task

KERN 在规则层采用四个核心抽象：

- **Recipe**：描述主动动作的匹配规则。智能体提出动作后，系统根据动作类型（`verb`）、目标选择约束（`selector`）与前置条件（`condition`）判断该动作是否成立，并生成后续效果。
- **Reaction**：描述环境事件的被动响应。某一事件发生后，系统根据 `on_event` 和前置条件触发对应规则。
- **Effect**：描述一次结构化的状态写操作，例如移动实体、修改属性、创建任务或推进状态。
- **Task**：描述跨 Tick 持续执行的行为。对那些不能在单个 Tick 内完成的动作，系统会将其转化为任务流持续推进。

这四个抽象共同构成了 KERN 的规则描述方式。主动行为通过 `Recipe` 进入系统，被动行为通过 `Reaction` 进入系统，但两者最终都会汇入统一的 `Effect` 执行链。

### 4.3 规则声明的示例

KERN 的规则并不是硬编码回调，而是声明式配置。例如，环境推进时间后，系统可以通过如下规则推进实体状态：

```json
{
    "on_event": "AdvanceTick",
    "condition": { "type": "has_component", "component": "StatusComponent" },
    "effects": [ { "effect": "StatusTick" } ]
}
```

这段配置表示：当实体收到 `AdvanceTick` 事件，且实体拥有 `StatusComponent` 时，系统生成一个 `StatusTick` 的 Effect 来推进其状态变化。通过这种方式，规则被从核心代码中抽离出来，转移为可维护的数据。

## 5. 执行机制

### 5.1 系统启动链路

系统启动后，首先读取运行配置并装载场景数据。随后，系统依次完成数据加载、规则描述与状态变更指令的合法性校验，以及世界状态构建，并得到统一的运行时状态容器 `WorldState`。只有在这一阶段通过校验的数据，才会进入后续 Tick 运行阶段。

这一设计的目的，是将“配置错误”尽可能前置到运行前，而不是等到仿真进行到中途时再暴露。

### 5.2 Tick 驱动的运行时循环

KERN 采用离散 Tick 推进环境。每个 Tick 内部遵循固定顺序：

1. 系统推进全局时间。
2. 向环境中的实体广播 `AdvanceTick` 事件。
3. 普通状态类被动响应规则（`Reaction`）根据事件生成对应的状态变更指令（`Effect`）。
4. 智能体在同一时间推进框架下执行感知与动作生成。
5. 动作经过交互引擎 `InteractionEngine` 匹配主动规则（`Recipe`）后，被翻译为即时状态变更或长期任务。
6. 所有状态变更指令汇入统一执行器，修改 `WorldState`，并写入日志与 checkpoint。

这种固定顺序的好处是，所有仿真过程都有一致的时间基准。动作耗时、任务推进与状态变化都以整数 Tick 为单位推进，因此同一输入可以得到可复现的结果。

### 5.3 Effect 指令的生命周期

在 KERN 中，状态更新不会直接对内存对象进行随意写入，而必须先经过统一的 Effect 生命周期。这是整个系统最核心的执行机制之一。

一条 Effect 从产生到落地，通常经历以下步骤：

1. **生成**：动作匹配 `Recipe`，或事件匹配 `Reaction`，生成原始状态变更指令。
2. **归一化与校验**：归一化与校验模块（`Binder`）对指令进行参数补全、类型检查和格式标准化。非法输入在这一阶段被拦截。
3. **实体解析与上下文合并**：指令中的抽象引用会被解析为真实实体或运行时上下文。
4. **执行器分发**：执行器根据指令类型将请求发送到具体执行模块（`Handler`）。
5. **状态落地与事件记录**：执行模块修改 `WorldState`，并派发对应事件写入日志系统。

这一设计的主要收益在于，世界状态具备统一写入口。相比“任意逻辑中直接修改对象字段”的做法，KERN 的方式更冗长，但更容易审计，也更利于定位错误来源。

### 5.4 边界防护

为了保证执行链的稳定性，KERN 在运行时加入了若干边界保护。当前版本已经实现的保护包括：

- `max_trigger_depth`：限制递归触发深度，避免规则互相触发造成死循环。
- `BindError` / `ExecutorError`：对于不合法的输入或执行失败，不直接破坏运行时，而是形成可追踪的错误记录。
- 统一执行入口：所有状态写入都要经过执行器，减少隐式副作用。

这部分并不是系统的宣传重点，但它对内核稳定性是必要的基础支撑。

## 6. 智能体接入边界

### 6.1 接入原则

KERN 当前并不将智能体工作流本身视为核心贡献。它更关注的是：在不预设智能体内部算法的前提下，为外部智能体提供稳定的输入输出边界。

因此，KERN 对智能体只规定两件事：

1. 智能体如何从当前运行时状态容器 `WorldState` 中获取可见信息。
2. 智能体如何将决策结果表达为动作命令并送回系统。

换句话说，KERN 约束的是接口，而不是内部思考过程。

### 6.2 基于容器嵌套层级的视野可见性计算

在感知阶段，KERN 并不会把整个世界状态直接暴露给智能体。系统首先根据空间拓扑与实体嵌套关系计算可见性。

例如：

- 若 Agent 位于“医务室”，则默认只能看到同属该空间的顶级实体。
- 若柜子、箱子等容器未显式声明为透明，系统不会递归展开其内部内容。
- 只有在容器可见或动作权限允许的情况下，内部实体才会进入感知结果。

这一机制的作用有两个：一是降低无意义的上下文输入，二是维持信息不对称场景中的物理边界。

### 6.3 语义与物理的隔离

在动作输出阶段，智能体可以给出一个动作 JSON，但这个动作不会直接生效。它首先进入交互引擎 `InteractionEngine` 接受语义层面的检查。

例如，当模型试图“拿起一个不可拾取的背景板”或“攻击一个并不在当前空间中的实体”时，系统会通过 `Recipe` 匹配和条件检查拦截这类动作。只有通过检查的动作，才会被翻译为底层 Effect 并进入执行器。

这种设计的目的，是把“智能体的语义规划”与“世界的物理结算”明确分离。前者可以继续迭代，后者必须保持稳定。对于大语言模型常见的动作幻觉问题，这种隔离可以防止错误规划直接污染底层状态。

## 7. 验证场景与当前证据

KERN 当前的验证目标，不是证明某一种智能体已经足够聪明，而是验证底层沙盒机制是否成立。为此，系统采用了两个不同复杂度的场景作为证据来源。

### 7.1 农场场景

农场场景用于验证低复杂度下的基础交互链路，包括：

- 实体移动
- 资源交换
- 配方触发
- 基础任务推进

这一场景的作用，是证明在简单生产与交易环境中，规则匹配与状态落地链路能够稳定运行。

### 7.2 太空狼人杀场景

太空狼人杀场景用于验证高复杂度下的信息不对称与复杂事件传播问题。该场景包含多角色目标冲突、跨房间事件传播以及视野受限条件下的交互。

这一场景的重点不是展示基线智能体的策略上限，而是验证：当系统面临更复杂的规则组合和更多不稳定输入时，底层物理结算、日志记录与动作拦截是否仍然可靠。

### 7.3 当前验证结论

当前版本已经能够证明以下几点：

- 同一套执行链可以支撑不同场景复用。
- 基础交互、状态推进与任务流可以在统一 Tick 框架下运行。
- 智能体即使输出不合理动作，也不会直接破坏底层世界状态。

当前版本尚未提供系统化的实验统计与对比指标，因此本文中的“验证”更适合理解为机制验证，而不是完整的实验评测。

## 8. 当前边界与后续方向

KERN 当前版本已经形成较稳定的内核骨架，但仍存在明确边界。

当前展示层更接近调试观察工具，而不是完整产品前端。其次，当前的智能体工作流主要承担基线验证作用，尚不代表该架构在策略质量上的上限。双场景验证已经能证明“同一内核、不同场景”的基本命题，但仍缺少更系统的实验记录与定量分析。
