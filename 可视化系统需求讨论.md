# 可视化系统需求讨论

## 1. 定位

这个可视化系统的定位不是“实时游戏前端”，而是一个：

- 基于 checkpoint 的只读播放器
- 面向演示、回放、调试、录屏、文章展示
- 以离散 tick 和事件驱动为核心

它的目标不是提供世界编辑能力，也不是直接控制模拟器运行，而是把模拟结果以更直观的方式播放出来。

## 2. 输入语义

当前系统的核心输入是 checkpoint 文件。

checkpoint 语义：

- `world` 表示当前 tick 结束时的完整世界状态
- `log` 表示当前 tick 发生的增量事件与交互

因此一个 tick 文件不是“过程帧”，而是“结果帧”。

这意味着：

- 不能直接把 `tick_T.json` 当成 tick T 的播放起点
- 正确播放逻辑应为：
  - 起始状态：`tick_(T-1)` 的 `world`
  - 播放事件：`tick_T` 的 `log`
  - 结束状态：`tick_T` 的 `world`

换句话说，播放器内部应该区分：

- `StateFrame(T)`：tick T 结束时状态
- `PlaybackSegment(T)`：从 `StateFrame(T-1)` 过渡到 `StateFrame(T)` 的过程

## 3. 播放模型

### 3.1 时间轴行为

时间轴应支持两种不同语义：

- 跳转
  - 直接切换到目标 tick 的结束状态
  - 不补播中间动画
  - 用于快速定位和查看状态

- 播放
  - 从当前 tick 前一帧稳定状态开始
  - 按当前 tick 的事件顺序逐个播放
  - 播放完成后进入当前 tick 的结束状态
  - 然后才能继续下一个 tick

### 3.2 播放顺序

播放一个 tick 的建议顺序：

1. 当前稳定画面停留在 `tick_(T-1)` 的状态
2. 读取 `tick_T.log`
3. 将 log 转换为一组可视事件
4. 按顺序播放：
   - agent 高亮/跳动
   - 气泡出现
   - 需要时的边上 travel 表现
   - 左侧日志追加
5. 所有事件播放完成
6. 画面状态切换为 `tick_T.world`

### 3.3 事件层级

建议不要直接让渲染层消费原始 `log`，而是增加中间层：

- 原始输入：`interaction` / `event`
- 中间结构：`VisualEvent`
- 渲染执行：`AnimationStep`

这样能把“数据结构”和“视觉表现”解耦。

## 4. 画布与布局原则

### 4.1 固定舞台画布

中间世界视图采用固定 16:9 设计画布：

- 设计尺寸：当前使用 `1200 x 675`
- 所有世界元素在这个统一坐标系中排布
- 外层容器只负责等比缩放和留白

设计目标：

- 无论屏幕尺寸、窗口大小、缩放比例如何变化
- 舞台内元素的相对位置和比例保持一致

这意味着：

- 不应将中间舞台全部改成 `vw/vh` 驱动
- 应坚持“固定坐标系 + 整体 scale”思路

### 4.2 留白策略

假设目标屏幕比例是 16:9。

因此：

- 16:9 屏幕：画面接近铺满
- 16:10 等比例：允许出现上下留白或局部白边

这是预期行为，不是问题。

## 5. 地图结构表现

### 5.1 地点

地点使用大圆节点表示。

每个地点节点当前显示：

- 地点名称
- 实体数量
- agent 槽位区
- 其他实体摘要区

### 5.2 路径

`world.paths` 表示地点之间的连线。

当前路径表现：

- 地点之间以线段连接
- 标签显示 `travel_type` 或 `distance`

后续目标：

- 尽量减少边的重合
- 支持边偏移、轻微弧线或并行线
- 为“travel 中的 agent 停留在边上”预留空间

### 5.3 手动布局

自动布局很有用，但无法保证所有图都好看。

因此系统应支持：

- 初始自动布局
- 用户手动拖拽节点微调
- 将布局坐标持久化到本地

当前已经采用：

- 自动圆环布局
- 节点拖拽
- localStorage 保存布局

后续继续保留这个方向。

## 6. 实体表现规则

### 6.1 Agent

agent 是重点表现对象。

当前要求：

- agent 使用圆形 token 表示
- 后续替换为圆形头像或图像
- agent 行动时应有明显演出反馈

后续表现目标：

- 跳动
- 高亮
- 头顶气泡
- travel 时在边上驻留

### 6.2 其他实体

非 agent 不需要和 agent 使用同等级表现。

当前建议：

- 仅保留摘要或简化 token
- 不让大量普通实体破坏舞台整体布局
- 超出数量时汇总为 `+N`

## 7. Travel 视觉语义

travel 是一个重点语义，因为它涉及“在路中”的状态表达。

目标表现：

- 角色在 travel 过程中，不应被放在起点或终点圆内
- 而应被放在地点连线的中间或偏移位置

因此后续需要两类 travel 语义：

### 7.1 事件级 travel

当当前 tick 的事件中出现 travel：

- 当前行动 agent 应沿边移动或出现在边上
- 作为一次演出事件

### 7.2 状态级 travel

如果 agent 当前任务仍然处于 travel 状态：

- agent 在稳定状态下也应显示在边上
- 而不是回到地点圆内部

这一点依赖：

- 当前任务状态
- travel 相关任务进度
- 或由 log 推导出的中间状态

## 8. 日志与播放关系

左侧日志不是静态列表，而应与播放过程联动。

理想行为：

- agent 执行动画
- 气泡弹出
- 然后左侧日志再出现对应条目

也就是说，日志列表应是“播放结果的一部分”，而不是一开始就全部渲染完。

建议做法：

- 稳定状态下可显示该 tick 已完成的 log
- 播放中按事件顺序逐条插入

## 9. 当前已实现内容

截至目前，已经完成的基础设施包括：

- checkpoint 语义调整
  - `world` 为当前 tick 结束状态
  - `log` 为当前 tick 的增量事件
- 单文件 HTML viewer
- 左侧日志 / 中间地图 / 右侧详情 / 底部 timeline 布局
- 地点图视图
- 自动圆环布局
- 节点拖拽与坐标持久化
- 固定 16:9 舞台画布
- 地点节点内部的舞台元件化
- agent 圆形 token 初版

## 10. 当前未完成但已达成共识的方向

以下需求已经讨论清楚，但尚未实现：

- 播放 / 暂停状态机
- 拖动时间轴时的“跳状态不补播”
- tick 逐事件回放
- `VisualEvent` 中间层
- agent 跳动动画
- agent 图像替换
- agent 头顶气泡
- 左侧日志按播放顺序插入
- travel 在边上的驻留与移动
- 边重叠优化

## 11. 推荐开发顺序

建议后续按以下顺序推进：

### 阶段 1：播放器状态机

先建立内部播放模型，而不是先做复杂动画。

需要明确：

- 当前稳定状态属于哪个 tick
- 当前正在播放哪个 tick
- 当前 tick 已播放到第几个 visual event
- 播放结束后如何推进到下一 tick
- 拖动时间轴时如何切换稳定状态

### 阶段 2：VisualEvent 层

把原始 log 转成可视事件。

例如：

- agent 行动
- 说话
- travel
- 任务完成
- 创建/移动/销毁实体

### 阶段 3：Agent 演出

先实现最能体现“活起来”的部分：

- 圆形头像
- 跳动
- 高亮
- 气泡

### 阶段 4：Travel 边上语义

在路径上显示：

- 临时 travel 动画
- 持续中的“在路上”状态

### 阶段 5：边优化与视觉细化

- 减少边重合
- 调整标签位置
- 节点视觉统一
- 背景素材映射

## 12. 明确不做的事情

为了避免范围失控，当前应明确不做：

- 实时模拟控制前端
- 场景编辑器
- 双向同步 UI
- 复杂游戏逻辑前端
- 大规模美术资源系统

当前目标始终是：

- 播放
- 回放
- 演示
- 分析

## 13. 一句话总结

这个可视化系统本质上是一个：

**基于 checkpoint 的、固定舞台坐标系的、事件驱动的世界回放播放器。**

它的核心难点不在渲染技术本身，而在于：

- 正确理解 tick 结束状态
- 正确处理 `上一帧状态 + 当前帧事件 -> 当前帧结果`
- 在这个基础上构建可控的播放状态机
